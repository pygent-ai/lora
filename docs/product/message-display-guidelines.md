# 消息展示效果规范

日期：2026-06-21

## 使用约定

后续凡是涉及聊天区、消息流、工具调用展示、流式输出、Markdown 渲染、消息气泡样式的问题，都先读这份文档，再决定实现方案。

这份文档记录的是我希望 Lora 桌面端消息展示最终呈现出来的效果。它优先级高于临时实现细节；如果代码当前做法和这里冲突，应以这里的产品体验为目标，并结合现有架构逐步调整。

## 总体目标

消息区要像一个本地 AI 工程工作台里的执行记录，而不是普通网页聊天窗口。

整体感觉应当是：

- 干净、克制、可扫描。
- 银白、浅灰、金属科技感为主。
- 信息密度适中，方便连续阅读和调试。
- 用户消息、助手回复、运行过程、工具结果都有清楚层级。
- 运行状态像仪器遥测，不像大面积彩色提醒。

不要把聊天区做成营销页、社交聊天 App、深色控制台，或大面积蓝紫科技风。

## 视觉语言

消息展示沿用桌面端的银白科技风：

- 背景以浅银灰为主，保持平整。
- 消息面板以白色、近白色、浅银灰为主。
- 金属感来自细边框、轻阴影、内高光、冷灰层次，而不是明显渐变。
- 状态色只用于小面积标记、文字、边框或状态点。
- 普通消息区域避免蓝色、紫色、青色大面积填充。

推荐基调：

- 聊天区背景：浅银灰。
- 助手消息：白底、细灰边框、左侧轻微钢灰强调线。
- 用户消息：浅银灰底、细灰边框、右对齐。
- 工具和运行消息：白底卡片，灰色边框，状态小标记。

## 消息类型

聊天区至少区分四类展示对象：

- 用户消息。
- 助手正文消息。
- 运行过程消息，包括 thinking、runtime activity、状态摘要。
- 工具调用消息，包括工具调用、工具结果、成功、失败、运行中状态。

这些对象在视觉上要互相有关联，但不能混成一种气泡。

用户消息是输入记录，应该更紧凑、更靠右。

助手消息是主要阅读对象，应该更宽、更稳定、更适合长文和 Markdown。

运行和工具消息是执行遥测，应该能折叠，默认不抢正文注意力，但展开后足够清楚。

## 用户消息

用户消息应当：

- 右对齐。
- 使用紧凑气泡。
- 不渲染 Markdown，按普通文本展示。
- 支持换行、长单词、路径、命令自动换行。
- 宽度不应铺满整行，避免把短输入放大成主内容。
- 使用浅银灰或近白背景，不使用强色块。

用户气泡的重点是“我刚才说了什么”，不是内容排版展示区。

## 助手消息

助手消息应当：

- 左对齐。
- 带有小而克制的 Lora 身份标识。
- 主体使用白色或近白色阅读面板。
- 支持 Markdown，但视觉上像技术笔记，不像浏览器文章。
- 行高舒适，适合阅读较长回答。
- 长内容稳定换行，不挤压布局。

助手消息是聊天区的核心内容，应当拥有最高阅读优先级。

助手正文不应该依赖一个巨大的富文本标签来完成所有渲染。理想方向是组件化块渲染：段落、标题、列表、引用、代码块、表格分别有稳定样式。

## Markdown 展示

Markdown 只服务于可读性，不追求完整网页排版。

优先支持：

- 标题。
- 段落。
- 无序列表和有序列表。
- 引用块。
- 行内代码。
- fenced code block。
- 表格。
- 分隔线。
- 链接。

标题要紧凑，不要像网页大标题。聊天区里的标题是阅读层级，不是页面 hero。

列表要有清楚缩进和适度间距，但不要撑得过松。

引用块应当像柔和 callout：浅灰底、左侧细线、文字偏 muted。

链接可以使用克制的蓝灰色，但不能让整个消息显得蓝。

## 代码块

代码块是高优先级展示对象。

代码块应当：

- 使用等宽字体。
- 保留缩进、换行和空格。
- 横向溢出时内部滚动，不撑破消息列。
- 背景稳定，边框清楚。
- 和普通段落有明确区分。
- 语言标签可显示，但要小而安静。

当前银白主题下，代码块可以用浅灰技术面板。若后续选择深色代码块，也必须保证和银白主界面协调，不能把整条助手消息变成深色控制台。

## 表格

表格应当：

- 外层可横向滚动。
- 表头有轻微浅灰背景。
- 单元格边框细而清楚。
- 小屏或窄列时不挤压整条消息。
- 内容换行时仍保持可读。

表格是信息呈现组件，不应被当作普通 Markdown 文本粗略塞进气泡。

## 运行过程消息

运行过程消息用于表达“系统正在做什么”，不是最终回答。

它应当：

- 默认比助手正文更低调。
- 可以折叠和展开。
- 折叠时是一行摘要。
- 展开时显示有结构的过程详情。
- running、success、error 状态清楚但克制。

运行中状态可以有轻微动效或状态点，但不要大面积闪烁或高饱和色。

完成后，如果没有必要，不应持续占用比正文更强的视觉注意力。

## 工具调用展示

工具调用要像执行遥测卡片。

折叠状态：

- 单行摘要。
- 左侧显示最近或最重要的工具动作描述。
- 右侧显示展开/收起图标。
- running 状态可轻微脉冲。
- success 使用小面积绿色表达。
- error 使用小面积红色表达。
- 不显示内部 tool id 作为主要文案。

展开状态：

- 使用一个组合卡片展示一组连续工具调用。
- 左侧可以有状态轨道或状态点。
- 右侧展示每个工具调用的名称、状态、时间、耗时、结果摘要。
- 多个工具调用保持顺序。
- 结果详情可滚动，长输出不能撑爆页面。

工具调用展示的重点是“我做了什么、成没成功、结果在哪里”，而不是把原始 JSON 直接暴露给用户。

## 流式输出

流式输出必须稳定。

助手回复 streaming 时：

- 当前助手消息固定在原位增长。
- 不应反复重建整个 transcript，导致滚动、闪烁或布局跳动。
- 未完成的 Markdown 要优雅降级。
- 未闭合代码块、列表、引用都不能造成整个消息样式崩坏。

当 final answer 到达时，可以替换为最终文本，但视觉位置和阅读连续性要保持。

如果同时存在 thinking 或工具调用，正文、过程、工具的顺序要符合真实事件顺序。

## 间距和尺寸

消息区要有呼吸感，但不能松散。

建议：

- transcript 内边距保持桌面工作台级别的舒适留白。
- 消息之间留出明确间距。
- 气泡内边距适中，正文不贴边。
- avatar 小而稳定，不抢注意力。
- 用户短消息不被拉满。
- 助手长消息允许占据较宽区域，便于阅读。

所有固定格式元素都要有稳定尺寸，避免 hover、状态变化、展开收起导致布局乱跳。

## 状态表达

状态表达遵循“小而明确”的原则。

可用状态：

- ready / idle：灰色或钢灰。
- running：钢灰状态点、chip 或轻微动效。
- success：绿色小标记、文字或边框。
- warning：黄色小标记、文字或边框。
- error：红色小标记、文字或边框。

不要用整块大面积纯色背景来表达状态，尤其不要让 success/error 把整条消息染成强烈色块。

状态不能只靠颜色，必要时要有文字或图标。

## 文案规则

消息展示中的文案要面向用户和调试者，而不是内部实现。

应该显示：

- 工具动作的自然语言描述。
- 运行状态。
- 可理解的错误摘要。
- 必要的时间和耗时。
- 可展开的细节。

避免显示：

- 裸露的内部 tool id。
- 未整理的原始事件名。
- 没有解释的 JSON。
- 对用户没有意义的框架字段。

原始数据可以放在展开详情或 inspector，不应成为默认主视觉。

## 可访问性和交互

消息展示必须保留：

- 文本可选择。
- 代码可复制。
- 链接可点击。
- 折叠控件可键盘访问。
- 图标按钮有 title 或 aria-label。
- 焦点态清楚。

不要为了视觉效果使用会破坏选择、复制和阅读的软件绘制方式。

## 响应式要求

桌面端优先，但窄宽度下也不能崩。

必须保证：

- 文本不溢出父容器。
- 长路径、长命令、长 URL 能换行或滚动。
- 表格和代码块内部滚动。
- 工具结果不会撑爆 transcript。
- 折叠侧栏或 trace 面板时，聊天消息仍然可读。

## 实现方向

后续实现消息展示时，优先采用组件化思路：

- `MessageRow` 只负责消息行。
- 用户消息、助手消息、活动消息、工具消息分别拆出组件。
- Markdown 渲染尽量使用明确的块结构。
- 工具调用状态单独建模，避免靠字符串拼接驱动 UI。
- 样式使用全局银白科技 token，不写一次性硬编码颜色。

如果某次任务只做局部调整，也要避免把临时逻辑继续堆进一个巨大的消息组件里。

## 验收标准

一次消息展示相关改动完成后，应检查：

- 用户消息右对齐、紧凑、可换行。
- 助手消息左对齐、可读、Markdown 稳定。
- 代码块、表格、引用块不会破坏布局。
- streaming 过程中不闪烁、不跳位、不丢内容。
- 工具调用可折叠，展开后顺序和状态清楚。
- running、success、error 状态克制且可辨认。
- 长输出、长 URL、长路径不会撑破界面。
- 文字可选择，代码可复制。
- 视觉上仍然是银白、浅灰、金属科技感，而不是蓝紫深色科技风。
